23. **[RESOLVED] INCONSISTENT CHANGE TRACKING**
    Status: ✓ FIXED (Verified 2026-01-26)
    Location: src/Hmm.Core.Dal.EF/Repositories/* and EfEntityLookup.cs
    Problem: Some queries used AsNoTracking(), some didn't
    Resolution: Standardized change tracking strategy documented and implemented
    Design Decision:
    - EfEntityLookup (explicit lookup service): Uses AsNoTracking() for all queries
      * GetEntityAsync<T>: All type branches use AsNoTracking()
      * GetQueryableEntities<T>: All type branches use AsNoTracking()
      * Purpose: Read-only lookups that won't be modified
    - Repository classes (data access layer): Keep tracking enabled by default
      * Reason: Repository results may be assigned to other entities and updated
      * Using AsNoTracking() in repositories causes tracking conflicts when
        entities are later used in Update() operations
      * The Update() method attaches entities, conflicting with untracked instances
    - For explicitly read-only operations in application code, use IEntityLookup
    Impact: Clear separation between read-only lookups (AsNoTracking) and
    repository operations (tracking enabled for update scenarios)
